# OBSICLAUDE v1.2.0 — Major Update Design Spec

## Overview

Transform OBSICLAUDE from a sidebar-only chat plugin into a full editor-integrated AI assistant. Add right-click AI menu, @ mentions, typing animation, status bar, improved home screen, and robustness fixes for long content generation.

**Target:** Community plugin store approval + competitive parity with Copilot/Text Generator UX.

---

## 1. Right-Click AI Menu (Editor Integration)

### What
Text selection → right-click → OBSICLAUDE submenu with 5 AI actions → result inserted directly into editor.

### Actions

| Action | Prompt sent to Claude | Result placement |
|--------|----------------------|-----------------|
| Summarize | "Summarize this text concisely:\n\n{selection}" | Below selection |
| Translate | "Translate to {targetLang}:\n\n{selection}" | Below selection |
| Improve | "Improve this text — better clarity, flow, grammar:\n\n{selection}" | **Replaces** selection |
| Explain | "Explain this in simple terms:\n\n{selection}" | Below selection |
| Ask Claude... | Opens Modal for custom prompt input | Below selection |

### Language detection for Translate
- If selection contains Korean (Unicode range `\uAC00-\uD7AF`) → translate to English
- Otherwise → translate to Korean
- Respects `settings.language` as fallback

### Implementation
- Register `editor-menu` event in `main.ts`
- New file `src/editor-actions.ts` handles all 5 actions
- Uses existing `ClaudeService.quickAction()` for API calls
- Editor manipulation via Obsidian's `Editor` API: `editor.replaceSelection()` and `editor.replaceRange()`

### Edge cases
- **No text selected:** Menu items are hidden (only show when selection exists)
- **Very long selection (>10,000 chars):** Truncate to first 10,000 chars with notice "Selection truncated for AI processing"
- **API error during action:** Show Notice with error, do NOT modify editor
- **User selects across multiple lines:** Works — selection is passed as-is
- **Read-only / preview mode:** Menu items hidden (editor not available)

### "Ask Claude..." Modal
- Simple Modal with single textarea + "Send" button
- Placeholder: "What should Claude do with this text?"
- Closes on send, result inserted below selection

## 2. @ Mentions in Chat

### What
Type `@` in chat input → fuzzy search popup for files/folders → selected item attached as context.

### Behavior
1. User types `@` in chat input
2. FuzzySuggestModal opens (reuse existing `FileFolderSuggestModal`)
3. User searches and selects file or folder
4. Selected item appears as chip in context bar (reuse existing `attachedContexts`)
5. User can type more `@` to add multiple items
6. On message send, all attached contexts are included

### Trigger
- Detect `@` character on input event
- Only trigger if `@` is at start of word (not inside a word like "email@test")
- Regex: `/(?:^|\\s)@$/` on current input value up to cursor position

### Implementation
- Add input event listener in `chat-view.ts` `onOpen()`
- On `@` detected → open `FileFolderSuggestModal`
- On selection → call existing `this.addContext()` method
- Remove the `@` character from input after selection

### Edge cases
- **@ inside a URL or email:** Don't trigger (check preceding chars)
- **Modal dismissed without selection:** Remove the `@` char, resume typing
- **Multiple @ in sequence:** Each opens a new modal, each adds a context chip
- **Vault with 10,000+ files:** FuzzySuggestModal handles this natively with virtualization

## 3. Typing Animation (Simulated Streaming)

### What
After API response arrives, display text word-by-word instead of all at once. Simulates streaming experience.

### Behavior
1. API call completes → full text received
2. Split text into words (by spaces)
3. Display words progressively at ~20ms intervals
4. Markdown is re-rendered at 80ms debounced intervals (reuse existing render logic)
5. During animation, scroll follows text
6. User can click "Skip" to show full text immediately

### Implementation
- New private method `animateText(text: string, contentEl: HTMLElement): Promise<void>`
- Called after `claudeService.chat()` returns, before final render
- Uses `setInterval` with cleanup on skip/abort
- Skip button appears during animation (small "Skip ⏭" link below text)

### Edge cases
- **Tool calls mid-loop:** Animation only applies to final text output, not intermediate tool status
- **User sends follow-up during animation:** Animation completes instantly, then processes follow-up
- **Very long text (5000+ words):** Increase interval speed to 10ms for long texts to avoid tedious wait
- **Abort (Escape):** Stops animation and shows full text immediately
- **Empty response:** No animation, show "No response" message

## 4. Status Bar

### What
Bottom status bar showing current model name and cumulative token usage. Click to switch model.

### Display
```
[Sonnet | 1,234 tokens]
```

### Behavior
- Shows on plugin load
- Token count updates after each API response (from `response.usage.output_tokens + input_tokens`)
- Click on model name → same model picker menu as header
- Token count resets on "New chat" (clear chat)
- Desktop only (status bar not available on mobile)

### Implementation
- `this.addStatusBarItem()` in `main.ts` `onload()`
- Store reference to update text after each API call
- Add click listener for model switching
- Expose update method for `ChatView` to call after responses

### Edge cases
- **Plugin loaded but chat not opened:** Show "Sonnet | —" (dash for no tokens yet)
- **Model changed mid-conversation:** Update immediately
- **Multiple conversations:** Token count is per-session, resets on new chat

## 7. Robustness: Long Content & Timeouts

### Problem
Long content generation (blog posts, research notes) fails silently because:
1. `max_tokens: 4096` is too low for tool calls containing long text
2. No API call timeout protection
3. `stop_reason: "max_tokens"` is silently ignored

### Solution: Token Limits

| Context | max_tokens | Rationale |
|---------|-----------|-----------|
| Default (settings) | 8192 | Already updated |
| Settings slider range | 1024 → **16384** | Allow users to go higher |
| Magic Write calls | **16384** (override) | Long-form content needs room |
| Auto-continue on truncation | Up to **3 continuations** | For very long content |

### Auto-Continue Logic
```
while (response.stop_reason === "max_tokens" && continuations < 3) {
  messages.push({ role: "assistant", content: response.content });
  messages.push({ role: "user", content: "Continue writing from exactly where you stopped. Do not repeat any content." });
  response = await this.callAPI(messages, tools);
  continuations++;
  // Merge text blocks
}
```

### Solution: Timeout Protection

```typescript
// Create timeout abort for each API call
const timeoutMs = isMagicWrite ? 300_000 : 180_000; // 5min for writes, 3min for chat
const timeoutId = setTimeout(() => {
  callbacks?.signal?.aborted || abortController.abort();
}, timeoutMs);

try {
  const response = await this.callAPI(messages, tools);
} finally {
  clearTimeout(timeoutId);
}
```

### Solution: User Feedback During Long Waits
- Processing indicator already shows elapsed time (existing feature ✅)
- Add phase labels: "Thinking..." → "Reading sources..." → "Writing..." → "Creating note..."
- If >30 seconds: show "This is a long response — please wait" in indicator
- If >120 seconds: show "Still working... (complex request)" in indicator

### Edge Cases

| Scenario | Handling |
|----------|---------|
| API returns 429 (rate limit) | Show "Rate limited. Please wait and try again." Notice |
| API returns 529 (overloaded) | Show "Claude is busy. Please try again in a moment." Notice |
| Network disconnected mid-call | requestUrl throws → caught by existing error handler → show error in chat |
| User aborts during long write | Existing Escape handler works → shows partial text if any |
| Response truncated mid-tool-call | Auto-continue sends "Continue" → Claude completes the tool call |
| 3 auto-continues still not enough | Show warning: "Response was very long and may be incomplete. Try breaking into smaller requests." |
| Empty response | Show "Claude returned an empty response. Please try again." |
| Malformed JSON in tool call | Catch JSON.parse error → log to console → show "Tool execution failed" in chat |

## 8. processFrontMatter() Migration

### What
Replace manual YAML string building with Obsidian's official `app.fileManager.processFrontMatter()` API.

### Why
- Current `objectToYaml()` in vault-tools.ts manually builds YAML strings
- This can produce invalid YAML and doesn't trigger MetadataCache updates
- `processFrontMatter()` is atomic, triggers cache updates, and handles edge cases

### Where to apply
- `set_frontmatter` tool → use `processFrontMatter()` instead of string manipulation
- `batch_frontmatter` tool → same
- `create_note` tool → keep manual YAML for initial creation (file doesn't exist yet), but validate format

### Frontmatter standards (from Obsidian docs)
- Use plural property names: `tags` (not `tag`), `aliases` (not `alias`), `cssclasses` (not `cssclass`)
- Dates in ISO 8601: `YYYY-MM-DD`
- Tags as YAML list without `#` prefix
